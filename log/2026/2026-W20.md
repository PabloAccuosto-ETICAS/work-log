## Wiselook audit

- **Partner API auth diagnostic + email to Roberto (Sat May 09).**
  Verified keys in `.env` against Roberto's email — match character
  by character. Discriminative test of auth headers via curl: the
  server processes `Authorization: Bearer` and rejects other formats
  with a clear message. Same call via Swagger UI returns HTTP 500
  instead of 401, with the same key. The `-i` flag on curl revealed
  both responses include `x-clerk-auth-*` headers indicating Clerk
  middleware processes the partner API key as a JWT and fails
  (expected — keys are not JWTs); behaviour after Clerk diverges
  by request origin (clean 401 via curl, crash via browser). Email
  sent to Roberto with curl reproducer + full response (CCs include
  Borja Gracia, new contact). Reply pending. Filter probes promoted
  to actionable-now while waiting.
- **Doc-discoverability protocol (Sat May 09).** Mid-session
  realisation: hours of partner-API hypothesising could have been
  short-circuited by reading
  [`docs/context/PARTNER-INTEGRATION-GUIDE.md`](https://github.com/Eticas-AI/wiselook-audit/blob/main/docs/context/PARTNER-INTEGRATION-GUIDE.md),
  which has been in the repo since 27 April and indexed in
  `STATE.md`. Three corrections applied to prevent recurrence:
  (1) inline references to relevant docs in each item of `Blocking on
  Wiselook` and `Can move now` of STATE.md, so the pointer sits next
  to the problem; (2) renamed the context-table section to
  `Read before hypothesising` with the rule literally as the heading
  and a short rationale prefacing the table; (3) new
  `Receiving new documentation` protocol in STATE.md (commit + entry
  in `docs/context/README.md` + row in the context table + inline
  reference in active items + INTERNAL_LOG entry) so future docs are
  reachable through three independent paths. Project-instructions
  block in the Claude UI updated with the rule.
